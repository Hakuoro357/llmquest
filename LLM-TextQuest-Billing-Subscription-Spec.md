# LLM-TextQuest Billing & Subscription Spec

## Назначение документа

Этот документ описывает биллинг и подписочную модель MVP `LLM-TextQuest`.

Его задача — зафиксировать:

- какие тарифы существуют;
- как определяется effective access пользователя;
- как работает покупка `Premium`;
- что происходит при успешной оплате, неуспешном списании, отмене и возврате на `Free`;
- как hidden caps связаны с подпиской;
- какие webhook-события и внутренние записи являются источником истины.

## 1. Принципы MVP-биллинга

Биллинг MVP строится по следующим правилам:

1. В продукте есть только два пользовательских плана:
   - `Free`
   - `Premium`
2. Есть только один платный продукт:
   - `premium_monthly`
3. Hidden caps применяются только на backend.
4. Billing logic не должна быть сложнее core gameplay.
5. Источником истины для доступа является backend, а не клиент и не Stripe-hosted UI.
6. В случае сомнений система должна выбирать более безопасное поведение для экономики.

Следствие:

- в MVP нет кредитов;
- нет годовой подписки;
- нет add-ons;
- нет pay-per-use;
- нет нескольких premium-уровней.

## 2. Тарифы и доступ

### 2.1 Тарифы

`Free`

- доступ к игре с hidden caps;
- ограниченное число игровых сообщений;
- ограниченное число миров;
- ограниченное число персонажей.

`Premium`

- увеличенные hidden caps;
- доступ к более высоким лимитам на сообщения, миры и персонажей;
- без отдельных add-on механик.

### 2.2 Effective access

Backend должен оперировать двумя связанными, но разными понятиями:

- `subscription status`
- `effective plan`

`subscription status` отражает платежное состояние подписки.

`effective plan` отражает фактический доступ пользователя в продукте.

Пример:

- пользователь может иметь запись подписки в статусе `CANCELED`, но effective plan остается `PREMIUM` до конца оплаченного периода;
- пользователь может иметь `PAST_DUE`, но effective plan еще не изменен до конца оплаченного периода, если это не противоречит бизнес-логике.

### 2.3 Источник истины по доступу

Источником истины для доступов является backend на основе:

- `User.plan`
- `UserSubscription.status`
- `UserSubscription.currentPeriodEnd`
- административных override-ов, если они есть

Клиент не должен самостоятельно вычислять право доступа.

## 3. Платежный провайдер

### 3.1 MVP-предположение

Для MVP используется:

- `Stripe Payments`
- `Stripe Billing`

### 3.2 Что Stripe делает

Stripe отвечает за:

- checkout;
- хранение метода оплаты;
- recurring billing;
- webhook-уведомления.

### 3.3 Что делает наш backend

Backend отвечает за:

- создание checkout session;
- прием и идемпотентную обработку webhook-ов;
- обновление `UserSubscription`;
- обновление `User.plan`;
- enforcement hidden caps;
- доступ к функциональности в продукте.

## 4. Billing-сущности

Billing опирается на сущности из [LLM-TextQuest-Data-Model-Spec.md](D:\pro\LMQuest\LLM-TextQuest-Data-Model-Spec.md):

- `User`
- `UserSubscription`
- `BillingEvent`
- `UserUsagePeriod`

### 4.1 Роли сущностей

`User`

- хранит текущий effective plan.

`UserSubscription`

- хранит текущее и историческое состояние подписки.

`BillingEvent`

- хранит входящие webhook-ы и их обработку.

`UserUsagePeriod`

- хранит usage для hidden caps и экономики.

## 5. Статусы подписки

Для MVP используются нормализованные статусы:

- `INACTIVE`
- `ACTIVE`
- `PAST_DUE`
- `CANCELED`
- `EXPIRED`
- `INCOMPLETE`

### 5.1 Семантика статусов

`INACTIVE`

- активной подписки нет;
- effective plan обычно `FREE`.

`ACTIVE`

- платежный статус нормальный;
- пользователь имеет `PREMIUM` на активный оплаченный период.

`PAST_DUE`

- возникла проблема со списанием;
- доступ определяется бизнес-правилом, описанным ниже.

`CANCELED`

- автопродление отключено или подписка отменена;
- до конца оплаченного периода effective plan может оставаться `PREMIUM`.

`EXPIRED`

- оплаченный период закончился;
- effective plan становится `FREE`, если нет override-а.

`INCOMPLETE`

- checkout начат, но подписка не активирована;
- effective plan `FREE`.

## 6. Hidden caps и тарифы

### 6.1 Что ограничивается

Hidden caps должны применяться к:

- игровым сообщениям/ходам;
- созданию миров;
- созданию персонажей.

### 6.2 Где проверяются лимиты

Проверка должна происходить на backend до:

- `POST /worlds`
- `POST /characters`
- `POST /sessions/:sessionId/turns`

### 6.3 Что видит пользователь

Пользователь может не видеть точные численные лимиты.

Система может показывать только состояния:

- `available`
- `near_limit`
- `blocked`

### 6.4 Что важно для биллинга

Billing и hidden caps нельзя рассматривать отдельно:

- подписка определяет effective plan;
- effective plan определяет cap policy;
- cap policy влияет на расход и экономику.

## 7. Основной purchase flow

### 7.1 Сценарий

1. Пользователь нажимает upgrade на `Premium`.
2. Клиент вызывает `POST /api/v1/billing/checkout-session`.
3. Backend создает Stripe checkout session.
4. Пользователь проходит оплату.
5. Stripe отправляет webhook.
6. Backend сохраняет `BillingEvent`.
7. Backend обновляет `UserSubscription`.
8. Backend обновляет `User.plan = PREMIUM`, если доступ должен быть активирован.

### 7.2 Важный принцип

Клиентский redirect после оплаты не должен быть единственным триггером активации доступа.

Источник истины:

- webhook + backend reconciliation.

### 7.3 Checkout success return

После возврата из Stripe клиент может показать статус:

- `payment_processing`
- `payment_confirmed`
- `payment_failed`

Но окончательное решение об access state остается за backend.

## 8. Поведение при успешной оплате

### 8.1 Первый успешный checkout

При первой успешной оплате backend должен:

- создать или обновить `UserSubscription`;
- записать `providerCustomerId`;
- записать `providerSubscriptionId`;
- поставить `status = ACTIVE`;
- записать `currentPeriodStart`;
- записать `currentPeriodEnd`;
- обновить `User.plan = PREMIUM`.

### 8.2 Эффект для продукта

После активации `Premium`:

- новые hidden caps применяются сразу;
- уже существующие игровые данные не меняются;
- новые ходы и создание сущностей проверяются по premium-лимитам.

---

## 9. Поведение при отмене подписки

### 9.1 MVP-правило

Для MVP отмена означает:

- `cancelAtPeriodEnd = true`
- пользователь сохраняет `Premium` до `currentPeriodEnd`
- после конца оплаченного периода backend переводит effective plan на `FREE`

### 9.2 Что не делать в MVP

Не нужно:

- мгновенно снимать доступ при обычной добровольной отмене;
- делать сложные pro-rated расчеты внутри продукта;
- вручную пересчитывать usage при отмене.

### 9.3 Что видит пользователь

Пользователь должен понимать:

- подписка отменена;
- доступ сохранится до конкретной даты конца периода.

---

## 10. Поведение при неуспешном списании

### 10.1 Первый платеж при checkout

Если первый платеж не прошел:

- подписка не должна активироваться;
- `User.plan` остается `FREE`;
- `UserSubscription.status` может быть `INCOMPLETE` или эквивалентный статус;
- пользователь не получает premium-access.

### 10.2 Неуспешное продление

Для MVP предлагается простое правило:

- если продление не удалось, статус становится `PAST_DUE`;
- доступ не продлевается сверх уже оплаченного периода;
- effective plan остается `PREMIUM` только до конца уже оплаченного периода;
- после истечения `currentPeriodEnd` effective plan становится `FREE`, если успешная оплата так и не произошла.

### 10.3 Почему так

Это правило:

- простое;
- безопасное для экономики;
- не требует сложной grace-period логики;
- не дает продукту месяцами обслуживать не платящего пользователя как premium.

### 10.4 Что видит пользователь

Пользователь должен видеть понятный статус:

- проблема с оплатой;
- доступ перестанет действовать после определенной даты, если оплата не будет исправлена.

## 11. Поведение при возврате на `Free`

### 11.1 Когда это происходит

Возврат на `Free` происходит, если:

- закончился оплаченный период отмененной подписки;
- истек период при `PAST_DUE`;
- подписка не была активирована;
- premium-access был отозван вручную администратором.

### 11.2 Что происходит с данными пользователя

При возврате на `Free`:

- пользователь не теряет миры, персонажей и историю автоматически;
- новые cap checks применяются уже по `Free`;
- система может блокировать создание новых сущностей или новых ходов, если usage/объем выше допустимого.

### 11.3 Пользователь, у которого уже "слишком много"

Если у пользователя на момент возврата на `Free`:

- больше миров, чем допускает `Free`;
- больше персонажей, чем допускает `Free`;

то для MVP рекомендуется правило:

- существующие сущности сохраняются;
- создавать новые сверх `Free`-ограничений нельзя;
- активное использование остается ограниченным правилами `Free`.

Это намного проще и безопаснее, чем принудительно удалять контент.

## 12. Refunds и disputes

### 12.1 MVP-подход

Refunds и disputes в MVP не должны становиться сложной автоматической системой.

Рекомендуемый подход:

- refund обрабатывается в Stripe;
- webhook записывается в `BillingEvent`;
- кейс помечается для ручной проверки, если он влияет на доступ;
- спорные случаи решаются через админку.

### 12.2 Правило доступа при refund

Для MVP лучше использовать консервативное правило:

- automatic instant access revocation после refund не обязателен;
- но администратор должен иметь возможность вручную снять premium-доступ, если refund подтвержден и это требуется политикой продукта.

### 12.3 Почему так

Потому что полностью автоматическая логика refund/dispute в MVP:

- усложняет систему;
- редко нужна на малом объеме;
- легко ломается в edge cases.

## 13. Webhook processing

### 13.1 Принципы

Webhook processing должен быть:

- идемпотентным;
- backend-only;
- журналируемым;
- устойчивым к повторной доставке.

### 13.2 Обязательные свойства

- каждый webhook сохраняется в `BillingEvent`;
- `providerEventId` уникален;
- повторная доставка не должна повторно менять доступ;
- ошибки обработки должны быть видны в monitoring и admin console.

### 13.3 Минимально важные события

На уровне бизнес-логики MVP особенно важны:

- checkout completed
- subscription created
- subscription updated
- subscription canceled
- invoice paid
- invoice payment failed

Нормализовать их следует в собственные статусы продукта, а не тащить Stripe-термины везде напрямую.

---

## 14. Admin override

### 14.1 Зачем нужен

Для MVP нужен ручной override, потому что без него support и ops будут застревать на billing edge cases.

### 14.2 Что должно быть возможно

Администратор должен иметь возможность:

- вручную выдать `Premium`;
- вручную вернуть на `Free`;
- вручную синхронизировать effective plan после сбоя webhook;
- оставить note о причине изменения.

### 14.3 Ограничения

Любой override должен:

- логироваться в audit log;
- быть доступен только admin-ролям;
- быть виден в карточке пользователя.

---

## 15. Экономические guardrails

### 15.1 Биллинг не должен ломать экономику

Для MVP важно зафиксировать:

- premium-access нельзя выдавать только на основании client redirect;
- hidden caps нельзя проверять на клиенте как источнике истины;
- prolonged grace periods без явной причины не нужны;
- failed payment не должен автоматически оставлять пользователя в premium дольше оплаченного периода.

### 15.2 Monitoring, связанный с биллингом

Нужно отслеживать:

- число активных premium-пользователей
- conversion в `Premium`
- cancel rate
- failed payment rate
- refund rate
- число admin overrides
- случаи рассинхронизации `User.plan` и `UserSubscription.status`

## 16. API-поведение для биллинга

### 16.1 Checkout

`POST /api/v1/billing/checkout-session`

- создает checkout session;
- требует auth;
- не должен активировать доступ сам по себе.

### 16.2 Subscription read

`GET /api/v1/billing/subscription`

- возвращает пользователю текущее состояние access и подписки;
- используется клиентом для статусов и billing UX.

### 16.3 Cancel

`POST /api/v1/billing/subscription/cancel`

- включает отмену по концу периода;
- не снимает доступ мгновенно при обычной добровольной отмене.

### 16.4 Webhook

`POST /api/v1/billing/webhook`

- принимает события провайдера;
- обновляет сущности backend;
- не зависит от участия клиента.

## 17. UX-правила для клиента

### 17.1 Что клиент должен уметь показывать

Минимально:

- current plan
- subscription status
- дата окончания доступа при отмене
- status проблемы с оплатой
- состояние hidden cap: `available / near_limit / blocked`

### 17.2 Чего клиент не должен делать

Клиент не должен:

- сам решать, premium ли пользователь;
- показывать точные hidden caps, если продукт решил их скрывать;
- активировать premium до подтверждения backend;
- строить доступ на основании одного только Stripe redirect.

## 18. Edge cases

### 18.1 Пользователь оплатил, но webhook задержался

Правило:

- клиент может показать `payment_processing`;
- backend должен дождаться подтверждения или периодически синхронизироваться;
- доступ не должен permanently зависать без recovery-механизма.

### 18.2 Webhook пришел повторно

Правило:

- повтор не должен менять доступ второй раз;
- событие должно считаться idempotent.

### 18.3 Пользователь отменил подписку и снова купил

Правило:

- новая подписка или реактивация должны корректно обновить `UserSubscription`;
- effective plan остается `PREMIUM`, если платеж подтвержден.

### 18.4 Пользователь дошел до hidden cap и сразу купил `Premium`

Правило:

- upgrade должен мгновенно снимать premium-ограничение после backend-confirmed activation;
- ранее заблокированная операция не обязана повторяться автоматически, но пользователь должен иметь возможность повторить действие без дополнительного friction.

---

## 19. Что можно отложить

На первый MVP можно не делать:

- годовые планы;
- trial period;
- coupons/promocodes;
- team/family billing;
- invoice dashboard для пользователя;
- сложный refund automation;
- multiple currencies;
- localized taxation logic внутри продукта beyond provider baseline.

---

## 20. Главный вывод

Billing MVP должен быть максимально простым, но жестким в двух местах:

- активация доступа;
- возврат на `Free` после окончания оплаченного периода.

Если коротко, правильная MVP-логика такая:

- один `Premium monthly`
- backend-enforced hidden caps
- webhook-driven source of truth
- cancel at period end
- no free premium after failed renewal beyond paid period
- no destructive deletion of worlds/characters on downgrade

Это дает системе предсказуемость, а экономике — шанс остаться под контролем.

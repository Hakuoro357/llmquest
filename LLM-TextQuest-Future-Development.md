# LLM-TextQuest - Дальнейшее развитие

## Назначение документа

Этот документ содержит функции и архитектурные идеи, которые не входят в MVP, но могут быть реализованы после подтверждения базовой ценности проекта. Он нужен как целевой roadmap и backlog для следующих фаз развития.

---

## 1. Продуктовые направления после MVP

После запуска MVP развитие проекта логично вести по четырём направлениям:

1. Углубление игровых механик.
2. Социальные и мультиплеерные функции.
3. Продвинутая LLM-архитектура и память.
4. Монетизация и премиальные возможности.

---

## 2. Расширение игровых механик

### 2.1 Режимы смерти

После MVP можно вернуть полную систему режимов смерти:

- `Обычный`
- `Хардкор`
- `Пермасмерть`

Дополнительно:

- предупреждения перед входом в рискованные режимы;
- отдельные правила для изгнания из мира;
- матрицу совместимости персонажей и миров.

### 2.2 Мир смерти

Полноценная система мира смерти может стать важной уникальной механикой проекта:

- персональный мир смерти для персонажа;
- испытание после гибели;
- возврат в сессию при успехе;
- временное изгнание при неудаче;
- повторные попытки через ритуал.

### 2.3 Ритуалы возврата

Возможности развития:

- требования к локации;
- требования к предметам;
- минимальное число участников;
- разные уровни сложности ритуала;
- последствия успеха, провала и критического провала.

### 2.4 Расширенные характеристики

После MVP можно поддержать:

- пользовательские системы характеристик;
- настройку правил мира под конкретный жанр;
- расширенные навыки, статусы, эффекты и свойства предметов.

## 3. Социальные и мультиплеерные функции

### 3.1 Кооперативные сессии

Следующий крупный этап после одиночной игры:

- сессии на 2-5 игроков;
- общий контекст сцены;
- синхронизация действий участников;
- отображение состояний всех персонажей.

### 3.2 Realtime-режим

После подтверждения спроса на мультиплеер:

- WebSocket или аналог для realtime-обновлений;
- доставка новых сцен и событий без перезагрузки;
- индикаторы активности участников;
- обработка конфликтов одновременных действий.

### 3.3 AFK-механика

Можно развивать текущую идею дальше:

- автоматическое narrative-объяснение бездействия;
- временное исключение персонажа из принятия решений;
- мягкое возвращение персонажа в активную сцену;
- отдельные правила AFK для боя и мирных сцен.

### 3.4 Друзья, приглашения, приватность

После MVP можно добавить:

- список друзей;
- заявки в друзья;
- приватные сессии по приглашению;
- совместные приватные миры;
- контроль доступа к контенту.

### 3.5 Присоединение к активной сессии

Сильная, но сложная функция:

- позднее подключение игрока к уже идущей истории;
- генерация сцены-перехода;
- автоматическое объяснение появления нового персонажа;
- summary для быстрого входа в контекст.

## 4. Продвинутая LLM-архитектура

### 4.1 Многоуровневая память

Полная версия системы контекста может включать:

- `Hot Context` в PostgreSQL hot tables или в отдельном cache-слое, если он действительно понадобится после MVP;
- `Warm Storage` в PostgreSQL с summary-блоками;
- `Cold Storage` для архива;
- meta-summary для очень длинных кампаний;
- semantically relevant recall через поиск по истории.

### 4.2 Векторный поиск

После MVP можно внедрить:

- embeddings для summary-блоков;
- выбор релевантных событий по текущему действию;
- восстановление давно забытых сюжетных линий;
- память о персонажах, фракциях и незавершённых конфликтах.

### 4.3 Разделение LLM-задач

Можно использовать разные пайплайны:

- одна модель для генерации мира;
- одна для сцен;
- одна для summary;
- одна для служебной нормализации и structured output.

Это поможет контролировать цену, задержку и качество.

### 4.4 Structured output и guardrails

После MVP стоит формализовать:

- JSON-схемы для сцен и обновлений состояния;
- валидацию ответа модели;
- fallback-обработку невалидных ответов;
- фильтры unsafe-контента;
- ограничения на самовольное изменение мира и состояния персонажа.

## 5. Монетизация и коммерческие функции

### 5.1 Premium-подписка

После MVP можно вводить:

- увеличенные лимиты миров и персонажей;
- больше активных сессий;
- продвинутая модель генерации;
- расширенная история;
- кастомизация профиля и интерфейса.

### 5.2 Кредиты и разовые покупки

Возможные сценарии:

- дополнительные слоты миров;
- дополнительные слоты персонажей;
- разовые premium-generation функции;
- косметические улучшения.

### 5.3 Пермасмерть как премиальная механика

Это может быть частью premium-пакета, но только после проверки спроса. До этого лучше считать её экспериментальной high-engagement функцией, а не основой монетизации.

---

## 6. Контент и модерация

После MVP потребуется отдельный контур безопасности:

- moderation пользовательских описаний миров;
- moderation пользовательских действий;
- ограничения по NSFW/насилию/токсичности;
- правила для публичных миров;
- жалобы и блокировки контента.

Это особенно важно до запуска публичных социальных функций.

## 7. Архитектурные улучшения

### 7.1 Выделение realtime-сервиса

Если нагрузка вырастет, имеет смысл вынести realtime-часть из основного Next.js-приложения.

### 7.2 Очереди и фоновые задачи

После MVP можно вынести в фон:

- генерацию summary;
- предгенерацию стартовых материалов мира;
- перерасчёт индексов памяти;
- отправку уведомлений.

### 7.3 Аналитика

Для развития продукта важно собирать:

- длину сессий;
- частоту возвратов;
- точки выхода;
- стоимость генерации;
- популярные типы миров;
- долю свободного ввода против выбора готовых действий.

## 9. Критерии перехода за пределы MVP

К следующим этапам стоит переходить только если:

- одиночная сессия удерживает пользователей;
- стоимость генерации контролируема;
- пользователи возвращаются в миры и к персонажам;
- базовая связность повествования уже стабильна;
- есть подтверждённый спрос на мультиплеер или сложные режимы.

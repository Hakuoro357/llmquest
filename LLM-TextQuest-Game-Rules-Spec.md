# LLM-TextQuest Game Rules Spec

## Назначение документа

Этот документ фиксирует формальные игровые правила для MVP `LLM-TextQuest`. Его задача — убрать двусмысленность между narrative-слоем и системным расчетом, чтобы backend, frontend и LLM-интеграция работали по одной и той же модели.

Документ описывает:

- базовые характеристики персонажа;
- шкалу сложности действий;
- механику проверок;
- правила `advantage/disadvantage`;
- типы исходов действий;
- `HP`, урон и состояние риска;
- правила одного хода в MVP;
- границы системы на первой версии.

---

## 1. Принципы MVP-правил

Правила MVP строятся вокруг четырех принципов:

1. Простота важнее полноты.
2. Игрок должен чувствовать закономерность, а не произвол.
3. `LLM` описывает результат, но не рассчитывает его.
4. Один ход должен обрабатываться быстро и предсказуемо.

Следствие:

- в MVP нет сложной RPG-математики;
- нет длинных цепочек модификаторов;
- нет скрытых подсистем боя, отличающихся от общей механики;
- почти все действия решаются через единый формат проверки.

---

## 2. Характеристики персонажа

В MVP используются три характеристики:

- `Физическое`
- `Ментальное`
- `Социальное`

### 2.1 Значения характеристик

На старте каждая характеристика имеет значение от `1` до `5`.

Стартовая схема MVP:

- одна сильная характеристика: `4`
- одна средняя характеристика: `3`
- одна слабая характеристика: `2`

При создании персонажа игрок:

1. выбирает одну сильную характеристику;
2. выбирает одну слабую характеристику;
3. оставшаяся характеристика автоматически получает среднее значение.

Пример:

- сильная `Физическое`, слабая `Социальное` -> итог: `Физическое 4`, `Ментальное 3`, `Социальное 2`
- сильная `Ментальное`, слабая `Физическое` -> итог: `Физическое 2`, `Ментальное 4`, `Социальное 3`

Такой подход делает старт персонажа понятным для игрока, сохраняет баланс и не требует ручного распределения чисел.

### 2.2 Модификаторы характеристик

Для всех системных проверок используется модификатор, вычисляемый из значения характеристики:

| Значение | Модификатор |
|----------|-------------|
| 1 | -2 |
| 2 | -1 |
| 3 | 0 |
| 4 | +1 |
| 5 | +2 |

В MVP рекомендуется не выходить за диапазон модификаторов `-2` ... `+2`.

### 2.3 Выбор характеристики для действия

Каждое действие игрока должно быть отнесено к одной основной характеристике:

- `Физическое`: сила, ловкость, выносливость, скрытность, преодоление препятствий, бой;
- `Ментальное`: наблюдательность, анализ, знание, планирование, взлом, магическое или логическое действие;
- `Социальное`: убеждение, обман, давление, дипломатия, чтение эмоций, влияние на NPC.

Если действие затрагивает несколько сфер, система выбирает одну главную характеристику, а не делает смешанную проверку.

### 2.4 Что нельзя решать через характеристики

Чтобы игрок не обходил слабые стороны персонажа, для каждой характеристики задаются ограничения.

Через `Физическое` нельзя:

- автоматически заменять анализ, знание или наблюдательность;
- решать сложные переговоры одной только демонстрацией силы, если сцена требует доверия, дипломатии или обмана;
- получать информацию, которой персонаж объективно не может владеть.

Через `Ментальное` нельзя:

- заменять физическое преодоление препятствий, если требуется сила, ловкость или выносливость;
- автоматически выигрывать социальные сцены только за счет "логичности" аргумента;
- отменять последствия опасной сцены без реального действия в мире.

Через `Социальное` нельзя:

- заменять наблюдательность, поиск следов, анализ механизмов или решение логических задач;
- проходить чисто физические испытания без подходящего действия;
- убеждать существ, которые не способны понимать речь, не расположены к контакту или находятся вне досягаемости влияния.

### 2.5 Правило честной слабости персонажа

Слабая характеристика персонажа должна реально ощущаться в игре.

Это означает:

- система не должна слишком легко переназначать проверку на более выгодную характеристику только из-за формулировки игрока;
- если сцена явно требует слабую сторону персонажа, эта слабость должна сохраняться как игровой фактор;
- игрок может искать обходной путь через тактику, предметы, подготовку или изменение ситуации, но не через словесное переименование действия.

Пример:

- если персонаж слаб в `Социальном`, он не должен стабильно превращать переговоры в `Ментальную` проверку только потому, что "говорит логично";
- если персонаж слаб в `Физическом`, он не должен проходить опасные физические препятствия только за счет описания "я делаю это аккуратно".

### 2.6 Архетипы персонажа

В MVP архетипы являются описательным слоем, а не отдельной механикой.

Архетип:

- помогает игроку быстрее понять роль персонажа;
- может использоваться в UX и генерации предыстории;
- не дает отдельной числовой системы вне характеристик.

Примеры архетипов:

- "Воинственный"
- "Наблюдательный"
- "Харизматичный"
- "Хитрый"
- "Упрямый"

## 3. Типы действий

Для MVP каждое действие должно быть интерпретировано как один из четырех базовых типов:

- `Исследование`
- `Социальное взаимодействие`
- `Рискованное физическое действие`
- `Использование предмета или особенности ситуации`

Тип действия нужен не для отдельной формулы, а для:

- выбора характеристики;
- расчета сложности;
- выбора возможных последствий;
- описания результата через LLM.

---

## 4. Правило одного хода

Один ход в MVP — это одна значимая попытка действия.

### 4.1 Что считается одним действием

Примеры корректного одного хода:

- "Осмотреть дверь и найти ловушку"
- "Убедить стражника пропустить меня"
- "Ударить волка факелом"
- "Использовать ключ на северной решетке"

Примеры слишком широкого действия:

- "Я убиваю всех врагов, хватаю артефакт и убегаю"
- "Я сначала договариваюсь с охраной, потом краду карту, потом сжигаю архив"

Если игрок вводит многошаговое действие, система:

1. выделяет первый значимый шаг;
2. сообщает или подразумевает, что остальные шаги потребуют следующих ходов;
3. обрабатывает только первую значимую попытку.

### 4.2 Обязательные этапы хода

Каждый ход проходит через следующие системные этапы:

1. Получение действия игрока.
2. Интерпретация намерения.
3. Определение типа действия.
4. Выбор характеристики.
5. Расчет `DC`.
6. Применение модификаторов ситуации.
7. Выполнение проверки `d20 + modifier`.
8. Определение исхода.
9. Обновление состояния.
10. Нарративный ответ LLM.

## 5. Основная проверка действия

Базовая формула MVP:

`d20 + модификатор характеристики + ситуативные модификаторы`

Результат сравнивается с `DC` действия.

### 5.1 Ситуативные модификаторы

В MVP рекомендуется ограничиться следующими значениями:

- `+1` умеренное преимущество от предмета, подготовки или окружения;
- `-1` умеренная помеха;
- `+2` сильное системное преимущество в редких случаях;
- `-2` сильная системная помеха в редких случаях.

Если ситуация уже учитывается через `advantage/disadvantage`, не стоит одновременно перегружать ее большими числовыми модификаторами.

### 5.2 Натуральные значения кубика

Для MVP рекомендуется использовать простое правило:

- натуральное `20` дает гарантированный успех с усиленным положительным эффектом;
- натуральное `1` дает гарантированный провал с осложнением.

Это правило помогает создать яркие моменты и делает систему более "настольной" по ощущениям.

### 5.3 Когда бросок не нужен

Не каждое действие требует проверки.

Проверка не нужна, если:

- действие логично, безопасно и не встречает значимого сопротивления;
- у игрока есть подходящий предмет или доступ, и сцена не создает давления;
- результат очевидно должен быть успешным по контексту;
- сцена не содержит значимой ставки.

Действие считается автоматически невозможным, если:

- оно противоречит установленным фактам мира;
- игрок пытается выполнить несколько крупных действий за один ход;
- действие физически, логически или социально недостижимо в текущей сцене;
- персонаж не имеет нужного доступа, ресурса или позиции, а попытка не может быть осмысленно разложена на более мелкий шаг.

В случае автоматической невозможности система не делает бросок, а возвращает игроку понятное narrative-объяснение и предлагает искать другой подход.

## 6. Шкала сложности DC

В MVP используется фиксированная шкала:

| DC | Значение |
|----|----------|
| 8 | Простое действие под давлением |
| 10 | Базовое осмысленное действие |
| 12 | Умеренно сложное действие |
| 14 | Сложное действие |
| 16 | Очень сложное действие |
| 18 | Редкое и опасное действие |
| 20 | Пограничное для MVP действие |

### 6.1 Практика выбора DC

Рекомендуемые правила:

- `DC 8-10`: игрок действует в подходящих условиях;
- `DC 12-14`: есть риск, ограничение времени или сопротивление;
- `DC 16-18`: значительный риск, сильный противник или сложная среда;
- `DC 20`: только для кульминационных или явно экстремальных ситуаций.

Для MVP желательно редко выходить выше `DC 16`, чтобы игра не ощущалась как постоянное наказание.

## 7. Advantage / Disadvantage

`Advantage` и `disadvantage` применяются вместо большого числа модификаторов, когда ситуация явно дает системное преимущество или системную помеху.

### 7.1 Правило применения

- `Advantage`: бросаются два `d20`, берется лучший результат.
- `Disadvantage`: бросаются два `d20`, берется худший результат.

### 7.2 Когда применять

`Advantage`:

- хорошая подготовка;
- сильное сюжетное преимущество;
- правильный инструмент;
- помощь среды или союзника;
- очевидно удачная тактика.

`Disadvantage`:

- ранения;
- нехватка времени;
- плохая видимость;
- неподходящий инструмент;
- действие в заведомо невыгодной позиции.

### 7.3 Ограничение для MVP

В одном ходе не должно быть более одного активного `advantage` или `disadvantage`.

Если одновременно есть основания и для преимущества, и для помехи, они взаимно отменяются.

## 8. Исходы проверки

В MVP у действия есть три базовых исхода:

- `Успех`
- `Частичный успех`
- `Провал`

### 8.0 Интерпретация результата проверки

Для системной стабильности рекомендуется использовать следующую схему:

| Результат относительно DC | Исход |
|---------------------------|-------|
| Натуральное `20` | Критический успех |
| `DC` и выше | Успех |
| `DC - 1` или `DC - 2` | Частичный успех, если сцена допускает компромисс |
| `DC - 3` и ниже | Провал |
| Натуральное `1` | Критический провал |

`Критический успех` и `критический провал` остаются частными случаями успеха и провала, но дают более яркое narrative-последствие.

### 8.1 Успех

Действие считается успешным, если итог проверки равен или превышает `DC` минимум на `0`, и при этом нет специальных условий для частичного исхода.

Результат:

- игрок достигает намерения;
- сцена продвигается вперед;
- система может дать дополнительную выгоду при высоком результате.

### 8.2 Частичный успех

Частичный успех нужен, чтобы игра не ломалась на каждом неидеальном броске.

Рекомендуемое правило для MVP:

- если итог проверки меньше `DC` на `1-2`, это `частичный успех`, если ситуация допускает компромисс;
- если действие по природе бинарное и компромисс невозможен, используется обычный провал.

Частичный успех чаще всего допустим для:

- социальных действий;
- исследования;
- скрытного продвижения;
- доступа к объекту через риск или цену;
- использования предмета в нестабильной ситуации.

Частичный успех обычно не применяется, если:

- действие полностью бинарно;
- нарушение логики сцены разрушает ставку эпизода;
- результат либо возможен, либо нет без промежуточного состояния.

Частичный успех означает:

- цель достигнута не полностью;
- или достигнута ценой осложнения;
- или достигнута с потерей ресурса, времени или позиции.

Примеры осложнений:

- шум привлек врагов;
- инструмент поврежден;
- NPC соглашается, но требует услугу;
- игрок получает доступ, но теряет время.

### 8.3 Провал

Провал происходит, если:

- результат ниже `DC` на `3+`;
- или действие бинарное и не допускает частичного успеха;
- или выпала натуральная `1`.

Провал должен двигать сцену вперед, а не просто останавливать игру.

Результат провала:

- ситуация ухудшается;
- появляется новый риск;
- игрок теряет ресурс, позицию или время;
- сцена переходит в более опасное состояние.

### 8.4 Допустимые типы осложнений

Чтобы осложнения были управляемыми, в MVP рекомендуется использовать ограниченный набор последствий:

- потеря времени;
- потеря позиции;
- повышение уровня угрозы;
- расход или повреждение предмета;
- ухудшение отношения NPC;
- блокировка простого пути и переход к более рискованному пути.

В одном ходе не должно быть более одной серьезной цены или одного крупного осложнения.

## 9. HP, урон и опасность

Для MVP используется простая система `HP`.

### 9.1 Стартовое HP

Рекомендуемое стартовое значение:

- `10 HP` для обычного персонажа.

Допустимо варьировать `HP` в зависимости от персонажа, но для MVP лучше использовать единое стартовое значение.

### 9.2 Когда применяется урон

Урон применяется, если:

- персонаж получает прямую физическую атаку;
- персонаж терпит опасный провал;
- среда наносит вред.

Источники урона в MVP:

- враги;
- ловушки;
- опасная среда.

Рискованный `partial success` сам по себе не должен наносить `HP`-урон. Его цена выражается через время, позицию, угрозу, ресурс или сюжетное осложнение.

### 9.3 Шкала урона

Для MVP достаточно трех уровней:

- `1 HP` — легкий урон;
- `2 HP` — заметный урон;
- `3 HP` — сильный урон.

В редких кульминационных случаях допустим `4 HP`, но это не должно быть нормой.

### 9.4 Пороговые состояния

Рекомендуемая логика:

- `7-10 HP`: нормальное состояние;
- `4-6 HP`: ранен, возможны narrative-эффекты и `disadvantage` в рискованных действиях;
- `1-3 HP`: критическое состояние, высокий риск тяжелых последствий;
- `0 HP`: персонаж выбывает из текущей сцены или терпит тяжелое поражение.

### 9.5 Что происходит при 0 HP

В MVP не используется полноценная система смерти мира.

При `0 HP` персонаж:

- проигрывает текущую сцену или эпизод;
- попадает в состояние поражения, пленения, потери сознания или иного narrative setback;
- не удаляется из игры;
- не умирает окончательно.

Это правило сохраняет ставки, но не ломает кампанию в первой версии.

### 9.6 Урон и поражение сцены — не одно и то же

В MVP важно различать:

- `урон`, который уменьшает `HP`;
- `поражение сцены`, которое ухудшает положение персонажа без обязательной потери `HP`.

Поражение сцены может означать:

- тревогу;
- провал переговоров;
- потерю доступа;
- пленение;
- потерю цели;
- вынужденное отступление;
- переход в худшую позицию.

Это позволяет сохранять разнообразие последствий и не превращать любую неудачу в чистое снижение `HP`.

### 9.7 Опасная сцена

Опасной считается сцена, в которой персонаж может получить прямой урон или тяжелое поражение из-за провала.

В MVP рекомендуется использовать три уровня:

- `Safe Scene` — угрозы прямого урона почти нет;
- `Risky Scene` — возможны осложнения и поражение сцены;
- `Danger Scene` — возможны урон, тяжелый провал, потеря сознания или выход из эпизода.

Эта классификация нужна для выбора последствий и должна определяться системой до расчета проверки.

### 9.8 Восстановление HP

В MVP `HP` может восстанавливаться следующими способами:

- заклинанием;
- предметом;
- зельем;
- отдыхом;
- сном.

Для первой версии рекомендуется простое правило:

- небольшое восстановление возможно внутри приключения через предметы, зелья или способности;
- надежное восстановление происходит во время отдыха или сна;
- конкретные численные значения восстановления можно зафиксировать позже на уровне контента и предметов.

## 10. Инвентарь и предметы

В MVP инвентарь должен быть простым.

### 10.1 Формат предметов

Каждый предмет может иметь:

- название;
- короткое описание;
- тип;
- опциональный ситуативный эффект.

### 10.2 Что может делать предмет

Предмет в MVP может:

- открывать доступ к действию;
- давать `+1` к проверке;
- давать основание для `advantage`;
- снижать риск осложнения;
- решать конкретную сюжетную задачу.

Предметы не должны тянуть за собой отдельную сложную подсистему прочности, редкости и множества числовых параметров.

---

## 11. Роль rules engine

`Rules engine` обязан решать только формальные вещи:

- выделить намерение;
- выбрать тип действия;
- выбрать характеристику;
- определить `DC`;
- применить модификаторы;
- сделать бросок;
- определить исход;
- обновить `HP`, состояние и структурные последствия.

`Rules engine` не должен:

- писать художественный текст;
- придумывать длинные сюжетные сцены;
- определять тон повествования;
- выдавать литературное описание результата.

---

## 12. Роль LLM

`LLM` должна работать только поверх рассчитанного результата.

LLM получает:

- контекст мира;
- состояние персонажа;
- последние события;
- действие игрока;
- структурный результат проверки.

LLM должна вернуть:

- краткое описание результата действия;
- новую сцену;
- 3-4 новых варианта действий;
- краткое narrative-summary для истории.

LLM не должна:

- менять рассчитанный исход;
- игнорировать `HP` и системные последствия;
- самовольно повышать или понижать сложность;
- подменять одно действие несколькими завершенными действиями.

---

## 13. Рекомендуемые поведенческие правила для сцены

Чтобы игровой цикл не разваливался, каждая сцена должна соблюдать следующие ограничения:

- один основной фокус сцены;
- одна локальная цель;
- ограниченный объем текста;
- ясные ставки;
- не более одного нового крупного осложнения за ход;
- варианты действий должны различаться по стратегии, а не только по формулировке.

Рекомендуемые типы вариантов действий:

- безопасный;
- прямой;
- хитрый;
- социальный или нестандартный.

### 13.1 Свободный ввод игрока

Свободный ввод остается обязательной частью MVP, но должен проходить нормализацию.

Система должна:

1. выделить намерение игрока;
2. проверить, не слишком ли действие широкое;
3. определить, возможен ли автоматический успех, проверка или автоматическая невозможность;
4. применить обычные правила одного хода.

Свободный ввод не должен обходить ограничения системы и не должен позволять завершать несколько ключевых действий за один ход.

### 13.2 Темп сцены

Один ход должен:

- менять локальную ситуацию;
- продвигать сцену вперед;
- не завершать автоматически крупную сюжетную арку без накопления предыдущих шагов.

Крупные цели должны достигаться через последовательность ходов, а не через одно универсальное действие.

---

## 14. Границы MVP

В этот документ сознательно не входят:

- классы;
- уровни;
- опыт;
- полноценная магическая система;
- отдельные правила боя и внебоевых сцен;
- спасброски разных типов;
- сложные статусы и долгосрочные дебаффы;
- смерть персонажа как финальный исход;
- пермасмерть;
- механика мира смерти.

Все это относится к следующим фазам развития.

## 15. Открытые решения, которые можно отложить

Следующие вопросы не блокируют старт MVP, если базовая схема выше уже принята:

- будет ли рост характеристик в рамках кампании;
- появятся ли архетипы персонажей;
- будет ли отдельная шкала воли, энергии или ресурсов;
- нужен ли легкий прогресс между сессиями;
- как именно оформлять поражение при `0 HP`.

Эти решения допустимо принять позже, если они не ломают основную формулу:

`один ход -> одна проверка -> один системный исход -> один narrative-ответ`

# LLM-TextQuest Turn Contract

## Назначение документа

Этот документ описывает технический и логический контракт одного хода в MVP `LLM-TextQuest`.

Его задача — зафиксировать единый формат взаимодействия между:

- клиентом;
- backend;
- rules engine;
- LLM-слоем;
- хранилищем истории сессии.

Документ отвечает на вопросы:

- что считается одним ходом;
- что отправляет клиент;
- как система обрабатывает действие;
- что рассчитывает rules engine;
- что получает и возвращает LLM;
- что сохраняется после хода.

## 1. Принцип одного хода

Один ход в MVP — это одна значимая попытка действия от игрока в рамках текущей сцены.

Ход не должен:

- включать несколько полноценных действий;
- завершать крупную сюжетную арку в одиночку без накопленного контекста;
- обходить ограничения системы через длинное описание.

Если игрок присылает слишком широкое действие, система должна:

1. выделить первый значимый шаг;
2. обработать только его;
3. остальные шаги оставить на следующие ходы.

---

## 2. Общая схема обработки хода

Каждый ход проходит через следующие этапы:

1. Клиент отправляет действие игрока.
2. Backend загружает текущее состояние сессии.
3. Система нормализует действие игрока.
4. Rules engine определяет:
   - допустимость действия;
   - тип действия;
   - основную характеристику;
   - необходимость проверки;
   - `DC` и модификаторы;
   - исход действия;
   - структурные последствия.
5. Формируется структурный payload для LLM.
6. LLM генерирует narrative-результат и следующую сцену.
7. Backend валидирует ответ LLM.
8. Система сохраняет событие, обновляет состояние сессии и возвращает ответ клиенту.

### 2.1 Архитектурная схема одного хода

Рекомендуемая архитектура MVP:

`Client -> Backend API -> Session Lock -> Input Normalizer -> Rules Engine -> Narrative Generator -> Response Validator -> Save State -> Unlock -> Client Response`

Смысл этой схемы:

- ввод игрока сначала нормализуется;
- игровая механика считается детерминированно на сервере;
- только после этого LLM превращает системный результат в narrative-ответ;
- блокировка сессии защищает от одновременной обработки нескольких ходов.

### 2.2 Разделение ответственности в архитектуре

В MVP должны существовать три разных слоя:

- `Input Normalizer`
- `Rules Engine`
- `Narrative Generator`

`Input Normalizer`:

- интерпретирует ввод игрока;
- выделяет основное намерение;
- определяет, не является ли действие слишком широким;
- возвращает structured output;
- не рассчитывает игровые исходы.

Для MVP рекомендуется использовать для `Input Normalizer` отдельную более быструю и дешевую модель, чем для narrative-генерации, если она стабильно справляется со structured output.

`Rules Engine`:

- реализуется как обычная серверная логика;
- не является отдельной моделью ИИ;
- выбирает характеристику;
- определяет `DC`;
- решает, нужен ли бросок;
- рассчитывает `success / partial_success / failure`;
- обновляет системное состояние.

`Narrative Generator`:

- получает уже рассчитанный исход;
- описывает результат действия;
- строит новую сцену;
- предлагает новые варианты действий;
- не имеет права менять системный outcome.

Эта архитектура обязательна для MVP, потому что она снижает произвольность и делает систему дебажимой.

## 3. Вход клиента

### 3.1 Минимальный payload хода

Клиент должен отправлять:

```json
{
  "sessionId": "session_123",
  "characterId": "char_456",
  "actionType": "preset_or_free",
  "actionText": "Я убеждаю стражника пропустить меня внутрь",
  "selectedOptionId": "opt_2"
}
```

### 3.2 Поля входа

- `sessionId` — идентификатор текущей сессии.
- `characterId` — идентификатор персонажа.
- `actionType` — тип ввода:
  - `preset`
  - `free`
  - `hybrid` при необходимости в будущем
- `actionText` — текст действия игрока.
- `selectedOptionId` — идентификатор готового варианта, если игрок выбрал предложенное действие.

Для MVP достаточно поддерживать:

- `preset`
- `free`

### 3.3 Валидация на входе

До запуска игровой логики backend должен проверить:

- существует ли сессия;
- принадлежит ли персонаж пользователю;
- активна ли сессия;
- не исчерпаны ли внутренние лимиты тарифа;
- не пустой ли `actionText`, если это свободный ввод;
- не отправляет ли игрок новый ход до завершения предыдущего.

## 4. Состояние, загружаемое перед ходом

Перед обработкой система должна загрузить:

- `World Context`
- текущее состояние персонажа
- текущее `HP`
- инвентарь
- текущую сцену
- локальную цель сцены
- последние 3-5 событий
- summary прошлых событий
- текущий уровень опасности сцены
- доступные варианты действий предыдущего ответа

---

## 5. Нормализация действия игрока

Перед расчетом система должна превратить сырой ввод игрока в нормализованную форму.

### 5.1 Цели нормализации

Нормализация нужна, чтобы:

- убрать двусмысленность;
- сократить слишком длинный ввод;
- выделить основное намерение;
- определить, не пытается ли игрок сделать несколько действий сразу.

### 5.2 Результат нормализации

Пример внутреннего представления:

```json
{
  "rawAction": "Я сначала отвлекаю стражника разговором, потом пробегаю внутрь и захлопываю дверь",
  "normalizedIntent": "Отвлечь стражника разговором",
  "isMultiStep": true,
  "ignoredRemainder": "потом пробегаю внутрь и захлопываю дверь"
}
```

### 5.3 Возможные результаты нормализации

- корректное одношаговое действие;
- многошаговое действие с выделением первого шага;
- неясный ввод, требующий fallback-интерпретации;
- невозможное или недопустимое действие.

### 5.4 Кто выполняет нормализацию

Для MVP нормализация должна выполняться отдельным lightweight LLM-шагом или аналогичным structured interpretation layer.

Важно:

- нормализация не является narrative-генерацией;
- нормализация не определяет финальный игровой исход;
- нормализация только подготавливает намерение игрока для rules engine.

Backend обязан валидировать результат нормализации перед передачей в rules engine.

Рекомендуемый архитектурный подход:

- для нормализации использовать более дешевую и быструю модель;
- для генерации сцены и narrative-ответа использовать более сильную модель;
- формальную игровую механику не отдавать модели вообще.

Такой подход снижает стоимость и задержку одного хода без потери качества narrative-слоя.

## 6. Решение rules engine

После нормализации rules engine должен выдать структурное решение.

### 6.1 Обязательные поля решения

```json
{
  "actionValidity": "valid",
  "resolutionMode": "check",
  "actionCategory": "social",
  "primaryStat": "social",
  "difficultyDC": 12,
  "statValue": 4,
  "statModifier": 1,
  "situationalModifier": 0,
  "rollMode": "normal",
  "rolls": [13],
  "finalRoll": 13,
  "totalResult": 14,
  "outcome": "success",
  "outcomeSeverity": "normal",
  "consequenceType": null,
  "hpChange": 0,
  "positionChange": "better",
  "threatChange": "none",
  "resourceChange": null,
  "notes": "Guard is convinced to allow passage."
}
```

### 6.2 Значения ключевых полей

`actionValidity`:

- `valid`
- `invalid`
- `impossible`

`resolutionMode`:

- `auto_success`
- `check`
- `auto_fail`

`actionCategory`:

- `exploration`
- `social`
- `physical_risk`
- `item_or_situation`

`primaryStat`:

- `physical`
- `mental`
- `social`

`rollMode`:

- `normal`
- `advantage`
- `disadvantage`
- `none`

`outcome`:

- `success`
- `partial_success`
- `failure`

`outcomeSeverity`:

- `normal`
- `critical_success`
- `critical_failure`

`consequenceType`:

- `time_loss`
- `position_loss`
- `threat_up`
- `resource_spent`
- `resource_damaged`
- `npc_attitude_down`
- `path_blocked`
- `scene_defeat`
- `hp_damage`
- `none`

### 6.3 Внутренняя природа rules engine

`Rules engine` в MVP не должен быть отдельной ИИ-моделью.

Он должен быть реализован как детерминированная серверная логика, например:

- набор TypeScript-функций;
- таблицы правил;
- helper-модули для выбора характеристики и `DC`;
- системные функции расчета броска и последствий.

Это нужно для того, чтобы:

- игровые исходы были предсказуемыми;
- систему можно было тестировать;
- баланс не зависел от поведения модели;
- отладка была возможна без анализа свободного текста.

ИИ может помогать интерпретировать ввод и описывать результат, но не должен заменять formal rules layer.

---

## 7. Когда rules engine не делает бросок

Rules engine может вернуть:

- `auto_success`
- `auto_fail`

### 7.1 Auto success

Используется, если:

- действие логично и безопасно;
- нет значимого сопротивления;
- у игрока уже есть все нужные условия;
- результат очевиден и не требует случайности.

### 7.2 Auto fail / impossible

Используется, если:

- действие невозможно в рамках сцены;
- оно противоречит миру или текущим фактам;
- игрок пытается сделать слишком много за один ход;
- у персонажа нет необходимых условий для такой попытки.

Для MVP желательно различать:

- `auto_fail` — попытка допустима, но заведомо не приводит к желаемому результату;
- `impossible` — действие вообще не может быть исполнено в текущих условиях.

---

## 8. Проверка и расчет исхода

Если `resolutionMode = check`, система обязана:

1. выбрать характеристику;
2. выбрать `DC`;
3. применить модификаторы;
4. выполнить бросок;
5. определить исход;
6. определить одно ключевое последствие.

### 8.1 Ограничение последствий

В одном ходе допускается:

- не более одного основного осложнения;
- не более одного основного `HP`-изменения;
- не более одного крупного изменения позиции.

Это ограничение не дает системе перегружать игрока последствиями.

---

## 9. Изменения состояния после rules engine

До вызова LLM система уже должна знать, что изменилось.

### 9.1 Какие поля состояния могут обновляться

- `hp`
- `inventory`
- `scene danger level`
- `npc attitude`
- `current objective`
- `position state`
- `threat state`
- `available paths`
- `session flags`

### 9.2 Что не должна менять LLM

LLM не должна самостоятельно:

- менять `HP`;
- переопределять outcome;
- добавлять второй несогласованный штраф;
- отменять невозможность действия;
- превращать частичный успех в полный успех.

## 10. Payload для LLM

LLM должна получать не сырую игру, а уже структурно рассчитанный контекст.

### 10.1 Рекомендуемая структура

```json
{
  "worldContext": {
    "title": "Замок Серых Ветров",
    "tone": "темное фэнтези",
    "keyFacts": ["В замке действует строгая стража", "Северный вход закрыт"]
  },
  "characterState": {
    "name": "Арен",
    "hp": 8,
    "stats": {
      "physical": 3,
      "mental": 2,
      "social": 4
    },
    "inventory": ["печать барона", "короткий кинжал"]
  },
  "sceneState": {
    "goal": "Попасть внутрь замка",
    "dangerLevel": "risky",
    "recentEvents": ["Герой дошел до ворот", "Стража требует объяснений"]
  },
  "playerAction": {
    "rawText": "Я убеждаю стражника пропустить меня внутрь",
    "normalizedIntent": "Убедить стражника пропустить внутрь"
  },
  "systemResolution": {
    "resolutionMode": "check",
    "actionCategory": "social",
    "primaryStat": "social",
    "dc": 12,
    "rollMode": "normal",
    "rolls": [13],
    "total": 14,
    "outcome": "success",
    "outcomeSeverity": "normal",
    "consequenceType": "none",
    "hpChange": 0
  }
}
```

### 10.2 Что обязательно должно быть в payload

- сжатый world context;
- состояние персонажа;
- текущая цель сцены;
- уровень опасности;
- действие игрока;
- структурный исход проверки;
- системные изменения состояния.

---

## 11. Ожидаемый ответ LLM

LLM должна вернуть ответ в структурированном формате.

### 11.1 Рекомендуемый формат

```json
{
  "resultText": "Стражник хмурится, изучает твою печать, а затем нехотя отступает в сторону.",
  "sceneText": "Ты проходишь во внутренний двор. Здесь темно, сыро и подозрительно тихо. Слева — казармы, справа — узкий ход к башне.",
  "choices": [
    {
      "id": "opt_1",
      "label": "Пойти к казармам"
    },
    {
      "id": "opt_2",
      "label": "Пробраться к башне скрытно"
    },
    {
      "id": "opt_3",
      "label": "Осмотреть двор"
    },
    {
      "id": "opt_4",
      "label": "Позвать стражника и расспросить его"
    }
  ],
  "eventSummary": "Герой убедил стражника и проник во внутренний двор замка.",
  "sceneGoal": "Найти безопасный путь к нужной башне"
}
```

### 11.2 Требования к ответу LLM

- `resultText` — коротко описывает исход только текущего действия;
- `sceneText` — задает новую локальную ситуацию;
- `choices` — 3-4 качественно разных варианта;
- `eventSummary` — краткая запись для истории;
- `sceneGoal` — локальная цель следующего шага.

### 11.3 Ограничения для ответа LLM

LLM не должна:

- пересчитывать механику;
- описывать дополнительные несогласованные последствия;
- пропускать несколько сцен вперед;
- завершать крупную арку без накопленного контекста;
- генерировать слишком длинный текст;
- выдавать варианты действий, не соответствующие новой сцене.

---

## 12. Валидация ответа LLM

После генерации backend должен проверить:

- все обязательные поля присутствуют;
- есть 3-4 варианта действий;
- `sceneText` и `resultText` не пустые;
- новый текст не противоречит system resolution;
- `sceneGoal` присутствует;
- длина текста не превышает допустимый лимит.

Если ответ LLM невалиден, система должна:

- попытаться выполнить один повторный запрос;
- либо использовать fallback-ответ на основе system resolution и текущего шаблона сцены.

---

## 13. Что сохраняется после хода

После успешной обработки система должна сохранить:

- сырой ввод игрока;
- нормализованный intent;
- решение rules engine;
- бросок и итог;
- narrative-результат;
- новую сцену;
- доступные варианты действий;
- summary события;
- обновленное состояние персонажа;
- обновленное состояние сессии.

## 15. Ответ клиенту

После завершения хода клиент должен получить:

- narrative-результат;
- новую сцену;
- варианты действий;
- текущую цель сцены;
- обновленное состояние персонажа;
- обновленный `HP`;
- признаки лимитов или блокировок, если они влияют на продолжение.

Пример:

```json
{

  },

    "hp": 8,
    "stats": {
      "physical": 3,
      "mental": 2,
      "social": 4
    }
  }
}
```

## 16. Ошибки и граничные случаи

Для MVP система должна отдельно обрабатывать:

- пустой ввод;
- слишком длинный ввод;
- попытку сделать несколько действий за один ход;
- невозможное действие;
- невалидный ответ LLM;
- превышение внутреннего лимита тарифа;
- повторную отправку хода до завершения предыдущего;
- ход в завершенной или неактивной сессии.

Рекомендуемый принцип:

- игрок всегда получает понятный ответ, а не "молчаливую" ошибку;
- системная ошибка не должна ломать сессию;
- при проблеме LLM игра должна иметь fallback-поведение.

### 16.1 Блокировка и последовательность ходов

Проверка "не отправляет ли игрок новый ход до завершения предыдущего" должна выполняться через состояние сессии.

Для MVP рекомендуется использовать у сессии следующие служебные поля:

- `status`
- `currentTurnId`
- `processingStartedAt`

Минимальные значения `status`:

- `idle`
- `processing_turn`
- `error`
- `finished`

Правило работы:

1. до начала обработки backend пытается атомарно перевести сессию из `idle` в `processing_turn`;
2. если переход успешен, начинается обработка хода;
3. если сессия уже в `processing_turn`, новый запрос не должен обрабатываться;
4. после успешного завершения или контролируемой ошибки статус возвращается в `idle` или переводится в `error`.

Это защищает от:

- двойной отправки действия;
- гонок между вкладками;
- повторных запросов из-за сетевых задержек;
- параллельной обработки двух ходов одной сессии.

## 17. Границы документа

В этот контракт пока не входят:

- мультиплеерные ходы;
- конкурентные действия нескольких игроков;
- AFK-механика;
- PvP;
- продвинутая анимация или streaming-ответы;
- сложные платежные сценарии;
- отдельный боевой режим с иной формулой.

Это относится к следующим фазам развития.
